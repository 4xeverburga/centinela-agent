# Logbook — 2026-05-10

## Entry 1 — Schema refactor: `inspections_queue`, composite PK, preprocessing at ingest

**Date**: 2026-05-10

### Changes

Renamed the `queue` table to `inspections_queue` and replaced the autoincrement `id` with a composite primary key `(file_id, system_version)`. Removed `sharpness_score` from the queue (preprocessing now happens before queueing). Added `SYSTEM_VERSION=may2026:0.1` to config.

Moved the sharpness check from `ProcessQueueItemService` to `IngestPhotoService`. If an image is blurry, it is rejected at ingest time — it never enters the queue. The `chat_history` row for that image is saved with `is_included_in_history=0` and `rejected_reason='blurry'`, so it is excluded from LLM context windows.

New columns on `chat_history`: `is_included_in_history INTEGER NOT NULL DEFAULT 1`, `rejected_reason TEXT NOT NULL DEFAULT ''`.

### HITL callback bug fix

Discovered a bug in the `/alertas` → callback flow: buttons were using `inspection.id` for `callback_data`, but `_handle_callback` treated the value as `review.id`. These are different auto-increment sequences, so the wrong `human_reviews` row was being updated.

**Fix**: `/alertas` now fetches pending `human_reviews` (trigger=`SUSPICIOUS_CATEGORY`, `answer=''`) instead of inspections. The button `callback_data` correctly uses `review.id`. For display, the corresponding `InspectionRecord` is looked up via `image_file_id`.

### HITL UX improvements

- Button labels: `✅ Confirmar alerta` / `❌ Descartar alerta`
- Question text: `¿Es correcta esta alerta?` / `Is this alert correct?`
- Post-click messages: `⚠️ Alerta confirmada por {reviewer}` / `✅ Alerta descartada por {reviewer}`

### Media group caption capture

`_handle_photo` now extracts `message.caption` and passes it to `IngestPhotoService`. The caption is stored in the `text` field of the `ChatMessage`. `build_chat_window_text` now filters by `m.text` (truthy) instead of `not m.file_id`, so photo captions are included in LLM context.

### `validated_by_admin` removal

Removed `validated_by_admin` from the `inspections` table and `InspectionRecord` entity. Review state is resolved exclusively through the `human_reviews` table. `get_pending_suspicious` now uses `NOT EXISTS (SELECT 1 FROM human_reviews WHERE image_file_id = i.image_file_id AND answer != '')`.

### `human_reviews` schema change

Renamed `queue_id INTEGER` → `image_file_id TEXT NOT NULL DEFAULT ''` to align with the new composite PK scheme (no more autoincrement queue IDs).

### Files modified (26 total)

Entities, DDL, 4 repositories (queue, history, inspection, human_review), 3 services (ingest_photo, process_queue_item, handle_hitl_response), 2 controllers (telegram_bot, queue_worker), vllm_llm_inspector, prompts (es/en), config, .env.example, __main__.py, 4 test fakes, 1 test file.

## Entry 2 — Data model alignment: composite PKs, FK chains, context anchoring

**Date**: 2026-05-10

### Composite PK chain

Aligned all three main tables around a consistent key scheme:

| Table | PK | FK to |
|---|---|---|
| `chat_history` | `(chat_id, message_id)` | `projects(project_id)` |
| `inspections_queue` | `(chat_id, message_id, system_version)` | `chat_history(chat_id, message_id)` |
| `inspections` | `(chat_id, message_id, system_version)` | `inspections_queue(chat_id, message_id, system_version)` |

- `chat_history`: dropped auto-increment `id`, PK is now `(chat_id, message_id)`.
- `inspections_queue`: PK changed from `(file_id, system_version)` → `(chat_id, message_id, system_version)`. Added FK to `chat_history`.
- `inspections`: dropped auto-increment `id`, PK is `(chat_id, message_id, system_version)` with FK to `inspections_queue`. Enforces 0-1 relationship (an inspection is the optional result of processing a queue item).

### `ChatMessage` — `chat_id` and `message_id` fields

`ChatMessage` now carries `chat_id: str` and `message_id: int` (from Telegram) as its first two fields. Both are persisted in `chat_history` and used as the natural key.

### Context window anchored by `ChatMessage`

`get_context_around(project_id, anchor: ChatMessage, max_messages, before_minutes, after_minutes)` replaces the old timestamp-based `get_recent_for_user`. Uses `message_id` for precise ordering (before/after the anchor) combined with timestamp for time-window cutoff. Per-side LIMIT prevents one side from starving the other.

### `QueueItem` — `message_id` field

`QueueItem` gains `message_id: int` so `ProcessQueueItemService` can look up the original `ChatMessage` via `get_by_message_id` and pass it as anchor.

### `ProcessQueueItemService.execute()` signature

Changed from `(file_id, system_version)` to `(chat_id, message_id, system_version)` — matches the queue PK.

### `is_representative` column removed

Removed from `inspections_queue`, `QueueItem`, and `ingest_photo`. Non-representative images in a media group never enter the queue — they stay in `chat_history` (with `is_included_in_history=0` if blurry).

### `InspectionRecord` — `chat_id` and `message_id` fields

`InspectionRecord` now carries `chat_id`, `message_id`, `system_version` as its first three fields (matching its PK). Dropped auto-increment `id`. `LLMInspector.inspect()` port updated to accept `chat_id` and `message_id`. `save()` return type changed from `int` to `None`.

### Media group caption inheritance

When a photo arrives with an empty caption and a non-empty `cluster_id` (media group), `ingest_photo` calls `get_caption_in_cluster(project_id, cluster_id)` to find the first message in the cluster with non-empty text and reuses it. This matches Telegram's behavior where only the first message in a media group carries the caption.
